# kpt Deployment Quick Reference

## Quick Commands

### Generate kpt Package
```bash
make kpt-package IMG=<registry>/<image>:<tag>
```

### Initialize and Deploy
```bash
cd kpt-package/
kpt live init --namespace focom-operator-system --inventory-id focom-operator
kpt live apply
```

### Check Status
```bash
kpt live status
```

### Remove Everything (Correct Order)
```bash
# 1. Remove ConfigSync first
kubectl delete -k config/configsync
kubectl delete secret gitea-secret -n config-management-system

# 2. Remove operator
cd kpt-package/
kpt live destroy
```

## Common Questions

### Q: Does `kpt live init` create the namespace?
**A: No.** The namespace is created when you run `kpt live apply` because the bundle includes the namespace definition.

### Q: What does `kpt live init` do?
**A:** It adds inventory tracking metadata to the Kptfile so kpt can track what resources it deployed.

### Q: Can I use both `make deploy` and `kpt live apply`?
**A: Not recommended.** They manage resources differently. Choose one method and stick with it.

### Q: What's the difference between `make kpt-package` and `make deploy`?
**A:**
- `make kpt-package` - Only generates the bundle file
- `make deploy` - Generates bundle AND deploys using kubectl

### Q: Do I need to run `kpt live init` every time?
**A: No.** Only once. After initialization, the Kptfile contains the inventory metadata.

## Workflow Comparison

### Development Workflow (make deploy)
```bash
# Build and deploy in one step
make docker-build docker-push deploy IMG=myregistry/focom-operator:latest

# Undeploy
make undeploy
```

### Production Workflow (kpt)
```bash
# Generate package
make kpt-package IMG=myregistry/focom-operator:v1.0.0

# Initialize (one-time)
cd kpt-package/
kpt live init --namespace focom-operator-system --inventory-id focom-operator

# Preview changes
kpt live apply --dry-run

# Deploy operator
kpt live apply

# Wait for operator
kubectl wait --for=condition=available --timeout=60s \
  deployment/focom-operator-controller-manager -n focom-operator-system

# Deploy ConfigSync
cd ..
kubectl get secret gitea-secret -n default -o yaml | \
  sed 's/namespace: default/namespace: config-management-system/' | \
  kubectl apply -f -
kubectl apply -k config/configsync

# Check status
cd kpt-package/
kpt live status
kubectl get rootsync focom-resources -n config-management-system

# Update (regenerate and reapply)
cd ..
make kpt-package IMG=myregistry/focom-operator:v1.1.0
cd kpt-package/
kpt live apply

# Remove everything (correct order: ConfigSync first, then operator)
kubectl delete -k ../config/configsync
kubectl delete secret gitea-secret -n config-management-system
sleep 5  # Wait for ConfigSync to clean up CRs
kpt live destroy
```

## Files Generated

```
kpt-package/
├── Kptfile                          # Package metadata (updated by kpt live init)
├── package-context.yaml             # Package configuration
└── focom-operator-bundle.yaml       # All operator resources (generated by make)
```

## Integration with ConfigSync

**⚠️ IMPORTANT:** The kpt package does NOT include ConfigSync. You must deploy it separately.

### Complete Deployment Sequence

```bash
# 1. Deploy operator with kpt
cd kpt-package/
kpt live apply

# 2. Wait for operator to be ready
kubectl wait --for=condition=available --timeout=60s \
  deployment/focom-operator-controller-manager -n focom-operator-system

# 3. Copy Git secret to config-management-system namespace
cd ..
kubectl get secret gitea-secret -n default -o yaml | \
  sed 's/namespace: default/namespace: config-management-system/' | \
  kubectl apply -f -

# 4. Deploy ConfigSync RootSync
kubectl apply -k config/configsync

# 5. Verify ConfigSync is syncing
kubectl get rootsync focom-resources -n config-management-system
```

### Why Separate?

- **Different lifecycles** - Operator vs GitOps configuration
- **Different namespaces** - Operator in `focom-operator-system`, ConfigSync in `config-management-system`
- **Flexibility** - You might want operator without ConfigSync for testing

## Undeployment

### Complete Removal (Recommended Order)

**⚠️ IMPORTANT:** Always remove ConfigSync BEFORE removing the operator to ensure clean CR deletion.

```bash
# Step 1: Remove ConfigSync RootSync
kubectl delete -k config/configsync

# Step 2: Remove Git secret from config-management-system
kubectl delete secret gitea-secret -n config-management-system

# Step 3: Wait for ConfigSync to clean up CRs
sleep 10

# Step 4: Remove operator with kpt
cd kpt-package/
kpt live destroy

# Step 5: Verify everything is gone
kubectl get namespace focom-operator-system  # Should error: NotFound
kubectl get rootsync -n config-management-system  # Should show: No resources found
kubectl get crd | grep -E "focom|provisioning.oran"  # Should show no results
```

### Why This Order?

1. **ConfigSync first** - Stops syncing and removes CRs it manages
2. **Operator second** - Removes CRDs (which would cascade delete any remaining CRs)
3. **Clean removal** - No orphaned resources or stuck finalizers

### Quick Undeployment Script

```bash
#!/bin/bash
# Save as: undeploy-kpt.sh

echo "Removing ConfigSync..."
kubectl delete -k config/configsync
kubectl delete secret gitea-secret -n config-management-system --ignore-not-found=true

echo "Waiting for ConfigSync cleanup..."
sleep 10

echo "Removing operator..."
cd kpt-package/
kpt live destroy

echo "Verifying cleanup..."
kubectl get namespace focom-operator-system 2>/dev/null && echo "⚠️ Namespace still exists" || echo "✅ Namespace removed"
kubectl get rootsync -n config-management-system 2>/dev/null | grep focom-resources && echo "⚠️ RootSync still exists" || echo "✅ RootSync removed"

echo "Done!"
```

### Partial Removal

**Remove only ConfigSync (keep operator):**
```bash
kubectl delete -k config/configsync
kubectl delete secret gitea-secret -n config-management-system
```

**Remove only operator (keep ConfigSync):**
```bash
cd kpt-package/
kpt live destroy
# Note: ConfigSync will continue running but won't have an operator
```

## Troubleshooting

### Error: package uninitialized
```bash
cd kpt-package/
kpt live init --namespace focom-operator-system --inventory-id focom-operator
```

### Error: inventory object already exists
```bash
# Force reinitialize
kpt live init --force --namespace focom-operator-system --inventory-id focom-operator
```

### Resources already exist
```bash
# Remove existing resources first
kubectl delete -f kpt-package/focom-operator-bundle.yaml
kpt live apply
```

### CRs stuck in Terminating state
```bash
# This happens if you remove operator before ConfigSync
# Remove finalizers manually
kubectl patch ocloud <name> -n <namespace> -p '{"metadata":{"finalizers":[]}}' --type=merge

# Or remove ConfigSync to let it clean up
kubectl delete -k config/configsync
```
